# Codex Workflow

这里的目标不是让 Codex 替你写完项目，而是让它成为一个高质量研究搭档。

## 1. 让 Codex 做仓库考古

适合参考项目初读。

```text
请阅读这个仓库的 README、目录结构和核心入口文件。
输出：
1. 模块地图
2. 数据流
3. 控制流
4. 最适合我手搓 1/10 版本时保留的设计
5. 暂时不需要学的复杂性
```

## 2. 让 Codex 做接口压力测试

适合你刚写完一版 API。

```text
我希望你不要改代码，先审查这个接口设计。
请列出：
1. 哪些状态不应该暴露
2. 哪些错误类型不够表达真实失败
3. 哪些 API 会导致未来难以测试
4. 你建议的最小修改
```

## 3. 让 Codex 生成测试，而不是功能

这是最适合学习的用法之一。

```text
请根据当前实现生成 property tests / fuzz cases。
重点覆盖：
1. 边界条件
2. 随机操作序列
3. crash 后恢复
4. 与一个 naive oracle 的对照
```

## 4. 让 Codex 做对照阅读

当你已经写出一版后，再读参考项目。

```text
请对比我的实现和参考项目的这个模块。
不要只说谁更好。
请解释：
1. 两者核心抽象差异
2. 性能差异来自哪里
3. 哪些 bug 在我的版本更容易出现
4. 哪些复杂性在 toy 版本里可以暂时忽略
```

## 5. 让 Codex 设计实验

适合你有多个方案时。

```text
我有 A/B 两种设计。
请不要直接下结论。
请帮我设计 3 个最小实验来区分它们：
1. 正确性实验
2. 性能实验
3. 可维护性或可观测性实验
每个实验都要说明预期信号。
```

## 6. 上下文管理

大型项目里，Codex 的关键能力不是一次读完整个仓库，而是管理上下文。

推荐每个项目维护：

- `notes/<project>/context.md`: 当前架构摘要
- `notes/<project>/questions.md`: 未解决问题
- `notes/<project>/decisions.md`: 设计决策记录
- `notes/<project>/experiments.md`: 实验结果

每次开始新会话时，先让 Codex 读这些文件，再进入代码。

## 7. 你应该亲手做的部分

这些环节最好自己写：

- 第一版核心数据结构
- 第一次 debug
- 每个 milestone 的复盘
- 架构图
- failure case 的解释

这些可以大量交给 Codex 辅助：

- 查文档
- 枚举边界条件
- 写测试样例
- 生成可视化脚手架
- 做代码审查
- 整理复盘文字

## 8. Dangerous But Useful Prompts

大胆实验时可以用：

```text
请提出 5 个会把这个实现打爆的测试。
每个测试都要解释它攻击了哪个 invariant。
```

```text
请把这个模块当成生产事故现场来审查。
假设线上已经出现数据损坏/死锁/性能崩溃，最可能的根因在哪里？
```

```text
请帮我写一个最小复现，而不是直接修 bug。
```
